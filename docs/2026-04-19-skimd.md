# skimd — Concept Doc

> **One-liner:** A cross-platform terminal markdown reader that makes READMEs pleasant to consume. Like `nano`, but for reading, and with proper markdown rendering.

---

## Why this exists

Reading a README from the terminal today means one of:

- `cat README.md` — raw markdown, no formatting
- `less README.md` — same raw markdown, just paged
- `glow README.md` — good, but Go-based, not always installed, and unfamiliar to most JS/Python devs
- Open it in a browser or editor — context switch out of the terminal

`skimd` aims to be the obvious answer: `npx skimd README.md` and you get a properly rendered, keyboard-navigable view. Zero install, sensible defaults, fast.

---

## Goals

- Render markdown faithfully in the terminal: headings, lists, tables, code blocks with syntax highlighting, links, blockquotes, inline code
- Snappy keyboard-driven navigation (vim-style + arrow keys)
- Optional TOC sidebar for jumping between sections
- Cross-shell support: Windows Terminal, PowerShell, modern cmd, macOS Terminal/iTerm, any Linux emulator
- Zero-install via `npx`, persistent install via `npm -g`
- Sub-100ms startup feel for typical READMEs (< 50KB)

## Non-goals (initial scope)

- Editing markdown — strictly read-only
- Rendering arbitrary HTML embedded in markdown (subset only, see open questions)
- Fetching remote URLs natively (piping from `curl` is fine)
- Mouse support — keyboard-first, mouse handling across shells is messy
- Themes / deep customization in v1 — sensible defaults, expose later

## Tech stack

| Layer | Choice | Why |
|---|---|---|
| Runtime | Node.js 20+ (LTS) | Top-level await, native fetch, broad availability |
| Language | TypeScript 5.x, strict | Fewer footguns, better refactor confidence |
| TUI framework | [Ink](https://github.com/vadimdemedes/ink) | React-for-CLIs, used by Claude Code, Gemini CLI, GitHub Copilot CLI; fast, mature, cross-platform |
| Markdown parser | [marked](https://github.com/markedjs/marked) | Fast, AST output, easy to walk |
| Syntax highlighting | [cli-highlight](https://github.com/felixfbecker/cli-highlight) | Lighter than shiki, good enough for v1 |
| CLI args | [meow](https://github.com/sindresorhus/meow) | Minimal, sensible defaults |
| Build | `tsup` | Single-file ESM bundle, fast cold start |
| Package manager (publishing) | npm | Broadest reach |

## Interaction model

### Keybindings

| Key | Action |
|---|---|
| `j` / `↓` | Scroll down one line |
| `k` / `↑` | Scroll up one line |
| `d` / `PageDown` | Scroll down half-page |
| `u` / `PageUp` | Scroll up half-page |
| `g` | Jump to top |
| `G` | Jump to bottom |
| `t` | Toggle TOC sidebar |
| `/` | Search forward |
| `n` / `N` | Next / previous match |
| `Tab` / `Shift+Tab` | Cycle through visible links |
| `Enter` | Open focused link in default browser |
| `?` | Help overlay |
| `q` / `Esc` / `Ctrl+C` | Quit |

### Layout

Two-pane when TOC is open:

```
┌──────────────────┬─────────────────────────────────────────┐
│ Table of         │ # README                                │
│ Contents         │                                         │
│                  │ This is the body, rendered with proper  │
│ ▸ Introduction   │ markdown formatting.                    │
│   Installation   │                                         │
│   Usage          │ ## Installation                         │
│   Configuration  │                                         │
│   Contributing   │     npm install -g skimd                │
│   License        │                                         │
│                  │                                         │
└──────────────────┴─────────────────────────────────────────┘
 skimd README.md     ↑↓ scroll  t toc  / search  ? help  q quit
```

Single-pane when TOC is closed or terminal width < 80 cols.

## Architecture

```
src/
  cli.ts                  # entry point, arg parsing, file resolution
  app.tsx                 # root Ink component, manages mode/state
  components/
    Reader.tsx            # main scrollable viewport
    TOC.tsx               # sidebar (collapsible)
    SearchBar.tsx         # / search input
    Help.tsx              # ? overlay
    StatusBar.tsx         # bottom hint line
    markdown/
      Heading.tsx
      Paragraph.tsx
      List.tsx
      CodeBlock.tsx
      Table.tsx
      Link.tsx
      Blockquote.tsx
      InlineCode.tsx
      ...
  markdown/
    parse.ts              # marked → AST
    render.ts             # AST → array of Ink nodes
    toc.ts                # extract heading tree
    search.ts             # text search across rendered lines
  hooks/
    useScroll.ts          # viewport position state
    useKeybindings.ts     # central key dispatch
    useTerminalSize.ts    # responsive layout
  util/
    terminal.ts           # detect width/height, color support
    open.ts               # cross-platform `open url` (uses `open` package)
    plainRender.ts        # non-TTY fallback rendering
```

### Data flow

1. `cli.ts` reads file, passes raw markdown string to `app.tsx`
2. `parse.ts` runs marked → AST (cached for the session)
3. `toc.ts` extracts heading tree from AST
4. `render.ts` walks AST, returns an array of pre-rendered "lines" (Ink elements with metadata: line index, link refs, headings)
5. `Reader.tsx` slices that array based on scroll offset and viewport height
6. Search builds a flat text index from rendered lines for `/` lookups

## MVP (v0.1.0)

Ship the smallest useful thing. Everything below is in scope; everything above (TOC, search, syntax highlighting, link opening, multi-file) is **out**.

1. CLI takes a single `.md` file path: `skimd README.md`
2. If no path given, default to `./README.md` in CWD
3. Renders markdown: headings, paragraphs, lists, code blocks (dim color, no syntax highlighting yet), inline code, links (colored text, not yet clickable), blockquotes
4. Scrollable with `j/k/↑/↓/PageUp/PageDown/g/G`
5. `q` / `Esc` / `Ctrl+C` to quit
6. `?` shows help overlay
7. Non-TTY fallback: if `!process.stdout.isTTY`, dump plain rendered text and exit (so `skimd README.md | less` and `skimd README.md > out.txt` still work)
8. Status bar at the bottom showing filename + key hints

**Definition of done for MVP:** install via npm, run on Windows Terminal + PowerShell + Bash on Linux + macOS Terminal, render this project's own README correctly.

## v0.2.0 — Polish

- Syntax highlighting in code blocks (`cli-highlight`)
- TOC sidebar (`t` to toggle)
- `/` search with `n` / `N` navigation
- Tab-cycle through visible links + `Enter` to open in browser
- Tables (marked supports them; rendering them in fixed-width is the work)

## v0.3.0 — Multi-file & discovery

- `skimd .` walks current dir, lists all top-level docs (`README*`, `CONTRIBUTING*`, `CHANGELOG*`, `LICENSE*`), shows a picker
- `skimd path/to/dir` same for any dir
- Respect `.gitignore` when discovering

## Future / stretch

- Theme support via `~/.config/skimd/config.json` or `$XDG_CONFIG_HOME/skimd/`
- Reading position memory per file (resume where you left off)
- `skimd <github-url>` fetches and reads a remote README
- Bookmarks within a doc (`m a` to mark, `' a` to jump)
- Inline images for terminals that support them (iTerm2, Kitty, WezTerm)

## Open questions

1. **Image handling.** Render `![alt](url)` as `[image: alt]` for v1, or detect terminal capability and use iTerm2 / Kitty inline image protocols where supported? Lean toward plain text in v1, capability detection in stretch.
2. **HTML in markdown.** GitHub READMEs often contain `<img>`, `<details>`, `<sub>`, `<picture>`, `<a>`, and badge-style links. Strip silently? Render text content only? Interpret a small subset? Lean toward "render text content, drop the rest" with a `--strict` flag for purists.
3. **Wide tables.** When a markdown table is wider than the terminal, do we (a) wrap cell contents, (b) horizontally scroll, or (c) fall back to a list-style rendering? Lean toward (a) for v0.2 with a note that (c) might come later for very narrow terminals.
4. **Link focus model.** With no mouse, what does "the current link" mean? Plan: `Tab` / `Shift+Tab` cycle through visible links with a highlight; `Enter` opens the focused one.
5. **Default file resolution.** `skimd` with no args — open `./README.md`, list all README candidates, or print usage? Lean toward `./README.md` as the 80% case, with a list-picker if multiple candidates exist (`README.md`, `README`, `readme.md`).
6. **Config location.** `~/.config/skimd/` (XDG-ish) or `~/.skimdrc` (older convention)? Lean XDG.

## Development guidelines for Claude Code

When building this with Claude Code, please:

- Start by scaffolding the directory tree above; commit empty stubs first
- Build MVP features in the order listed under v0.1.0 — each item should be a separate commit
- Use Ink's `useInput` hook for keybindings, centralized in `useKeybindings.ts` (don't scatter `useInput` across components)
- Keep the markdown AST → render layer pure and testable; UI components should receive pre-rendered nodes, not parse markdown themselves
- Add Vitest from the start; aim for solid coverage on `markdown/render.ts` and `markdown/search.ts`
- Test on at least one Windows shell (Windows Terminal + pwsh) before declaring MVP done — this is the highest-risk environment
- No mouse handling, no fancy animations, no theme system in MVP — resist scope creep
